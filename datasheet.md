# Tenacious-Bench v0.1 — Datasheet

*Following the Gebru et al. (2018) datasheet format.*

---

## 1. Motivation

**Why was this dataset created?**
Tenacious-Bench v0.1 was created to fill a measurement gap: existing benchmarks (τ²-Bench retail, BrowseComp, WebArena) grade generic multi-turn task completion but do not measure behaviors that are uniquely costly in B2B sales contexts—signal grounding, capacity honesty, tone preservation, consent-first coordination, and gap framing. This dataset provides machine-verifiable evaluation tasks for those five dimensions.

**Who created it?**
The Tenacious engineering team (Week 11, April 2026). The primary artifact author is listed in `metadata.author_model`.

**Who funded it?**
Internal R&D budget under the Tenacious AI agent project. No external funding.

---

## 2. Composition

| Field | Value |
|-------|-------|
| Total tasks | 250 |
| Dimensions | 5 (signal_grounding, capacity_honesty, tone_preservation, consent_coordination, gap_framing) |
| Difficulty levels | 4 (easy, medium, hard, adversarial) |
| Source modes | 4 (trace_derived, programmatic, multi_llm_synthesis, hand_authored) |
| Partitions | train (125), dev (75), held_out (50) |
| Format | Individual JSON files per task, schema-validated |
| Schema | `schema_tenacious_bench.json` |
| Contamination check | Passed (cosine < 0.85, no 8-gram exact matches vs `eval/trace_log.jsonl`) |

**What does each instance represent?**
Each task is a synthetic B2B sales scenario. An instance includes:
- A hiring signal brief (company, open role, signal confidence, source).
- A bench summary snapshot (available engineering roles by stack).
- A prospect context (company size, sector, AI maturity, thread turn count).
- A ground truth rubric (expected + forbidden behaviors).
- A rubric (max_score, pass_threshold, dimensions scored).
- Null `candidate_output` (populated at eval time).

**Are there recommended data splits?**
Yes: `train/` (50%) for fine-tuning, `dev/` (30%) for development/ablations, `held_out/` (20%, 50 tasks) as sealed evaluation slice. The held-out split must not be used during training or hyperparameter search.

**Are there any known errors or sources of noise?**
- Programmatic sweep tasks share fixture data from a limited pool (7 prospect contexts, 4 bench snapshots). This limits lexical diversity within each dimension-difficulty cell.
- Adversarial tasks are constructed by parameter mutation, not by LLM adversarial generation; they may not capture the full distribution of real adversarial inputs.

---

## 3. Collection Process

**How was the data collected?**
Tasks were generated by `generation_scripts/generate_dataset.py` via a deterministic parametric sweep (seed=42). The sweep crosses 5 dimensions × 4 difficulties × 4 source modes using fixture pools for hiring signals, prospect contexts, and bench snapshots derived from `seed/` and `schemas/` directories.

**Over what timeframe?**
Generated 2026-04-29.

**Were any ethical review processes conducted?**
The dataset contains no personally identifiable information (PII). All company names are synthetic. The dataset does not include real prospect data.

---

## 4. Preprocessing / Cleaning

**Was any preprocessing / cleaning / labeling applied?**
- Each task JSON was validated against `schema_tenacious_bench.json`.
- Contamination check run against `eval/trace_log.jsonl` (the Week 10 trace log). Result: CLEAN (0 violations).
- 30 tasks from the dev split were dual hand-labeled. Inter-rater agreement: 90% (κ = 0.78). Three rubric clarifications applied. See `inter_rater_agreement.md`.

---

## 5. Uses

**Has the dataset been used for any tasks already?**
Yes: as the evaluation target for the SimPO preference-tuning experiment described in `training/train_simpo.py` and `ablations/ablation_results.json`.

**What (other) tasks could the dataset be used for?**
- Evaluation of any B2B sales agent against the five Tenacious dimensions.
- Few-shot prompt engineering: the ground truth rubrics serve as exemplars.
- Training a discriminative judge/scorer for automated quality control.
- Regression testing after agent updates.

**Is there anything about the composition of the dataset or the way it was collected and preprocessed / cleaned / labeled that might impact future uses?**
- Tasks are grounded in Tenacious-specific business rules (bench_summary.json format, style guide). Applying this benchmark to a different sales agent would require adapting the fixture pools.
- The held-out split must not be released publicly before the final evaluation run is complete.

## 7. Maintenance

**Who maintains the dataset?**
The Tenacious engineering team.

**How can the owner / curator / manager be contacted?**
Via the project GitHub repository: `samuellachisa/tenacious-agent`.

**Will the dataset be updated?**
v0.2 is planned to add multi-LLM-synthesised tasks and trace-derived tasks from live agent runs. Contamination check will be re-run on each version increment.

**If others want to extend / augment / build on / contribute to the dataset, is there a mechanism for them to do so?**
PRs welcome on the GitHub repository. All contributions must pass the contamination check and achieve ≥ 80% inter-rater agreement on a 10-task sample before merge.
