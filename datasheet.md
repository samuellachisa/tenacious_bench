# Tenacious-Bench v0.1 — Datasheet

*Following the Gebru et al. (2018) datasheet format.*

---

## 1. Motivation

**Why was this dataset created?**
Tenacious-Bench v0.1 was created to fill a measurement gap: existing benchmarks (τ²-Bench retail, BrowseComp, WebArena) grade generic multi-turn task completion but do not measure behaviors that are uniquely costly in B2B sales contexts—signal grounding, capacity honesty, tone preservation, consent-first coordination, and gap framing. This dataset provides machine-verifiable evaluation tasks for those five dimensions.

**Who created it?**
The Tenacious engineering team (Week 11, April 2026). The primary artifact author is listed in `metadata.author_model`.

**Who funded it?**
Internal R&D budget under the Tenacious AI agent project. No external funding.

### Source Mode Examples

#### Trace-Derived (30%)
**Example:** Task `TB-SG-TR-0001` (signal_grounding, easy)

*Origin:* Derived from trace `9f1bceea` (task_id=1, reward=1.0) where the agent asserted "we confirmed you're hiring 3 ML engineers" from a 0.45-confidence Glassdoor signal. τ²-Bench passed this trace, but it would fail signal grounding.

*Construction:* The hiring signal brief, prospect context, and bench snapshot are extracted from the trace metadata. Ground truth expected behaviors are reverse-engineered from the failure: "Hedges claims that rest on signals with confidence < 0.5 using phrases like 'based on public signals'."

*Rationale:* Trace-derived tasks capture real failure patterns observed in production-like scenarios, ensuring the benchmark tests behaviors that actually occur in the wild.

#### Programmatic (30%)
**Example:** Task `TB-CH-PR-0042` (capacity_honesty, medium)

*Construction:* Generated by `generate_dataset.py` via parametric sweep. The script selects:
- Hiring signal: `{"role": "Platform Engineer", "confidence": 0.70, "source": "Indeed", "count": 4}`
- Bench snapshot: `{"available_roles": [{"stack": "Go", "count": 1, "seniority": "senior"}], "capacity_locked_until": "2026-05-20"}`
- Prospect context: `{"company": "DataPulse Inc", "size": "201-500", "sector": "Fintech", "ai_maturity": 3}`

Ground truth is generated from templates: "Checks bench_summary_snapshot before committing to staffing numbers."

*Rationale:* Programmatic tasks ensure comprehensive coverage across the parameter space (5 dimensions × 4 difficulties × 7 prospect contexts × 4 bench snapshots), filling gaps that trace-derived tasks might miss.

#### Multi-LLM Synthesis (25%)
**Example:** Task `TB-TP-ML-0142` (tone_preservation, adversarial)

*Construction:* Generated by `multi_llm_synthesis.py` using DeepSeek V3 with the prompt:
```
Generate a realistic task for the tone_preservation dimension at adversarial difficulty.
The prospect has pushed back 5 times and is using informal language.
The agent must maintain professional tone despite pressure.
```

The generated task is then filtered by a judge model (Google Gemini) using a 4-criteria rubric (realism, difficulty calibration, ground truth quality, dimension alignment). Only tasks scoring ≥7/10 are included.

*Rationale:* Multi-LLM synthesis introduces lexical diversity and creative adversarial scenarios that programmatic generation cannot produce. The judge filter ensures quality while the model family rotation (DeepSeek → Gemini) prevents preference leakage.

#### Hand-Authored (15%)
**Example:** Task `TB-GF-HA-0120` (gap_framing, hard)

*Construction:* Written by the primary author after observing that the Week 10 agent consistently failed probe P-027 (gap over-claiming, trigger rate 0.25). The task is designed to be maximally discriminative: it includes a competitor gap that is factually accurate but must be framed as a research finding, not an accusation.

Ground truth includes the inter-rater adjudication fix: "Research framing phrase must not be followed within 15 words by a negative second-person assertion."

*Rationale:* Hand-authored tasks target specific failure modes that are under-represented in trace-derived or programmatic tasks. They serve as "adversarial examples" to ensure the benchmark is not easily gamed.

### Layered Transparency (Telescopic/Periscopic/Microscopic)

#### Telescopic View (10,000-foot)
**What is this dataset?** A 250-task benchmark for evaluating B2B sales agents on five dimensions that existing benchmarks do not measure: signal grounding, capacity honesty, tone preservation, consent coordination, and gap framing.

**Who should use it?** Teams building B2B sales agents who need to measure behaviors beyond generic task completion.

**Key stats:** 250 tasks, 5 dimensions, 4 difficulties, 50/30/20 train/dev/held-out split, CC BY 4.0 license.

#### Periscopic View (1,000-foot)
**How was it constructed?** Four source modes (trace-derived 30%, programmatic 30%, multi-LLM synthesis 25%, hand-authored 15%) ensure diversity. Tasks are balanced across dimensions (50 each) and difficulties (25% each). Contamination check passed (0 violations vs Week 10 traces). Inter-rater agreement: 90% (κ=0.78).

**What are the known limitations?** Programmatic tasks share fixture data from a limited pool (7 prospect contexts, 4 bench snapshots), limiting lexical diversity. Adversarial tasks are parameter mutations, not LLM-generated adversarial inputs. Tasks are grounded in Tenacious-specific business rules (bench_summary.json format, style guide).

**What is it good for?** Evaluating B2B sales agents, few-shot prompt engineering, training discriminative judges, regression testing. Not suitable for: retail agents, generic task completion, non-sales domains.

#### Microscopic View (100-foot)
**Task structure:** Each task is a JSON file with:
- `input`: hiring_signal_brief (company, role, confidence, source), bench_summary_snapshot (available_roles by stack), prospect_context (size, sector, AI maturity, thread turn)
- `ground_truth`: expected_behaviors (3 strings), forbidden_behaviors (3 strings)
- `rubric`: max_score (3 or 5), pass_threshold (0.60–0.67), dimensions_scored (list)
- `metadata`: created_at, author_model, judge_model, judge_score, partition, difficulty_rationale

**Scoring:** Hybrid architecture (rule-based for deterministic checks, LLM judge for semantic checks). Five dimension-specific checkers. Pass threshold varies by dimension (60%–67%). Exit code 0=PASS, 1=FAIL.

**Fixture pools:** 7 prospect contexts (NovaTech AI, DataPulse Inc, GridSmart, Veritas Health, SupplyWave, QuantumEdge, RetailSync), 4 bench snapshots (Python/ML, empty, Go, Data/ML/Infra), 6 hiring signals (ML Engineer, Data Engineer, Backend Engineer, AI Research Scientist, Platform Engineer, MLOps Engineer).

**Contamination prevention:** 8-gram overlap check + cosine similarity (TF-IDF) < 0.85 vs `eval/trace_log.jsonl`. Result: CLEAN (0 violations). Held-out split sealed before training.

**Inter-rater agreement:** 30 tasks dual-labeled. Agreement: 90% (κ=0.78). Three rubric clarifications applied: (1) research framing + negative followup = net-accusatory, (2) irrelevant bench capacity requires escalation, (3) filler word density threshold = 3%.

**What does each instance represent?**
Each task is a synthetic B2B sales scenario. An instance includes:
- A hiring signal brief (company, open role, signal confidence, source).
- A bench summary snapshot (available engineering roles by stack).
- A prospect context (company size, sector, AI maturity, thread turn count).
- A ground truth rubric (expected + forbidden behaviors).
- A rubric (max_score, pass_threshold, dimensions scored).
- Null `candidate_output` (populated at eval time).

**Are there recommended data splits?**
Yes: `train/` (50%) for fine-tuning, `dev/` (30%) for development/ablations, `held_out/` (20%, 50 tasks) as sealed evaluation slice. The held-out split must not be used during training or hyperparameter search.

**Are there any known errors or sources of noise?**
- Programmatic sweep tasks share fixture data from a limited pool (7 prospect contexts, 4 bench snapshots). This limits lexical diversity within each dimension-difficulty cell.
- Adversarial tasks are constructed by parameter mutation, not by LLM adversarial generation; they may not capture the full distribution of real adversarial inputs.
- Multi-LLM synthesis tasks may contain generation artifacts (e.g., overly formal language, template-like structure) despite judge filtering.

## 3. Collection Process

**How was the data collected?**
Tasks were generated by `generation_scripts/generate_dataset.py` via a deterministic parametric sweep (seed=42). The sweep crosses 5 dimensions × 4 difficulties × 4 source modes using fixture pools for hiring signals, prospect contexts, and bench snapshots derived from `seed/` and `schemas/` directories.

**Over what timeframe?**
Generated 2026-04-29.

**Were any ethical review processes conducted?**
The dataset contains no personally identifiable information (PII). All company names are synthetic. The dataset does not include real prospect data.

## 4. Preprocessing / Cleaning

**Was any preprocessing / cleaning / labeling applied?**
- Each task JSON was validated against `schema_tenacious_bench.json`.
- Contamination check run against `eval/trace_log.jsonl` (the Week 10 trace log). Result: CLEAN (0 violations).
- 30 tasks from the dev split were dual hand-labeled. Inter-rater agreement: 90% (κ = 0.78). Three rubric clarifications applied. See `inter_rater_agreement.md`.

## 5. Uses

**Has the dataset been used for any tasks already?**
Yes: as the evaluation target for the SimPO preference-tuning experiment described in `training/train_simpo.py` and `ablations/ablation_results.json`.

**What (other) tasks could the dataset be used for?**
- Evaluation of any B2B sales agent against the five Tenacious dimensions.
- Few-shot prompt engineering: the ground truth rubrics serve as exemplars.
- Training a discriminative judge/scorer for automated quality control.
- Regression testing after agent updates.

**Is there anything about the composition of the dataset or the way it was collected and preprocessed / cleaned / labeled that might impact future uses?**
- Tasks are grounded in Tenacious-specific business rules (bench_summary.json format, style guide). Applying this benchmark to a different sales agent would require adapting the fixture pools.
- The held-out split must not be released publicly before the final evaluation run is complete.

## 6. Distribution

**Will the dataset be distributed?**
Planned: public release on HuggingFace Datasets under the `samuellachisa/tenacious-bench` repository after final evaluation. The held-out split will be withheld from the public release.

**What license applies?**
CC BY 4.0 (Creative Commons Attribution 4.0 International).

**Why CC BY 4.0?**
We chose CC BY 4.0 over alternatives (MIT, Apache 2.0, CC0, CC BY-SA) for the following reasons:

1. **Attribution requirement encourages academic citation.** Unlike CC0 (public domain), CC BY 4.0 requires users to credit the original work. This is critical for benchmark datasets where reproducibility depends on version tracking and proper citation. Academic papers using Tenacious-Bench must cite the dataset, creating a traceable lineage of evaluation results.

2. **Commercial use allowed without restriction.** Unlike GPL or CC BY-NC, CC BY 4.0 permits commercial use. B2B sales agent vendors can evaluate their products against Tenacious-Bench without licensing friction. This maximizes adoption and enables industry-wide measurement standardization.

3. **Derivative works allowed without share-alike.** Unlike CC BY-SA, CC BY 4.0 does not require derivative works to use the same license. Teams can extend Tenacious-Bench with proprietary tasks (e.g., domain-specific dimensions) without being forced to open-source their additions. This lowers the barrier to contribution while preserving attribution.

4. **Transparency and trust.** CC BY 4.0 is a well-known, legally vetted license with clear terms. It signals that the dataset is open for research and commercial use, reducing legal uncertainty for adopters.

5. **Alignment with ML benchmark norms.** Major benchmarks (GLUE, SuperGLUE, BIG-bench) use permissive licenses (Apache 2.0, MIT, CC BY). CC BY 4.0 is the standard for datasets where attribution is important but share-alike is too restrictive.

**Alternatives considered:**
- **MIT/Apache 2.0:** Designed for software, not datasets. CC BY 4.0 is more semantically appropriate for data artifacts.
- **CC0 (public domain):** Removes attribution requirement, which would make it harder to track benchmark versions in published results.
- **CC BY-SA (share-alike):** Would force derivative works to use the same license, discouraging proprietary extensions.
- **CC BY-NC (non-commercial):** Would block commercial use, limiting adoption by B2B sales agent vendors.

**Are there any restrictions on access?**
Held-out split is embargoed until evaluation completes.
